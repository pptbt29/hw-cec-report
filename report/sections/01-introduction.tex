\subsection*{1.1 研究背景与应用动机}
\addcontentsline{toc}{subsection}{1.1 研究背景与应用动机}

大语言模型推理正在从云端集中式服务，走向端侧、边缘和云端共同参与的协同服务形态。过去几年中，LLM inference 优化大体形成了两条成熟路线：一条是 on-device inference，通过压缩、量化、小模型和端侧 NPU/GPU 加速，使模型尽可能靠近用户运行；另一条是 cloud / datacenter LLM serving，通过高性能 kernel、continuous batching、KV Cache 管理、模型并行和多实例调度，在云侧 GPU 集群中提高吞吐、降低尾延迟并摊薄服务成本。已有 LLM serving 系统与综述工作分别从 runtime、memory management、batching/scheduling 和 distributed serving 等角度总结了这些机制 \refciterange{4}{8}, \refciterange{73}{75}。

这两条路线分别解决了局部问题，却都难以单独覆盖真实业务中的全部约束。端侧推理具有低时延和数据本地性优势，但受算力、内存、功耗和散热限制，很难长期承载复杂大模型和高并发服务。云侧推理资源充足、框架成熟，却必须经过网络访问，端到端时延、带宽消耗、隐私风险和服务路径依赖都会进入业务关键路径。与此同时，接入边缘、区域边缘和近用户算力节点中存在大量分散、异构且动态变化的资源，它们既无法简单等同于端侧设备，也不同于数据中心中的同构 GPU 集群。

CEC（Collaborative Edge Computing，协作边缘计算）正是在这种背景下成为 LLM inference 的重要系统形态。如图~\ref{fig:collaborative-edge-computing} 所示，CEC 将泛在、地理分布的端侧与边缘设备组织为可协作的资源池，并通过 Wi-Fi 接入点、路侧单元、5G 基站和网关等基础设施连接中心云与多类终端，共同完成计算任务。CEC 并不是简单地把模型从云端搬到边缘，而是把端侧、边缘和中心云视为一个多层协同资源池，使推理请求能够根据模型规模、请求复杂度、网络状态、节点负载、隐私要求和 QoS 目标动态选择模型形态、执行位置和服务路径。端-边-云协同智能研究已经指出，多层系统的关键在于资源、链路和服务位置之间的协同，而不是单点计算能力的替换 \refcite{13}。因此，CEC LLM inference 的核心问题不再只是“如何让模型跑得更快”，而是如何在异构、分布式、动态变化的端-边-云环境中，以可控成本满足业务时延、吞吐、服务连续性和数据隐私要求。

\subsection*{1.2 CEC 场景下边缘 AI 推理的关键挑战}
\addcontentsline{toc}{subsection}{1.2 CEC 场景下边缘 AI 推理的关键挑战}

CEC 场景下的边缘 AI 推理首先受到资源异构和碎片化的约束。端侧设备、接入边缘、区域边缘和中心云在 CPU、GPU、NPU、内存、显存、功耗预算和加速库支持上差异显著，模型部署不能沿用中心云中“节点规格相近、互联稳定、控制面统一”的假设。同一个模型在不同硬件上可能面临完全不同的显存瓶颈、算子支持和通信代价，部署策略也必须随资源状态动态调整。

LLM 推理本身还具有显著的状态和内存压力。Prefill 阶段会为后续 decode 生成 KV Cache，decode 阶段又会随着输出长度不断追加状态；在长上下文、多轮会话和高并发场景中，KV Cache、prefix cache 和 session state 往往成为比单次计算更难管理的运行时资源。一旦这些状态跨节点访问、迁移或重算，通信成本、服务连续性和隐私边界都会同时受到影响。

请求动态性进一步放大了调度难度。用户并发数、prompt 长度、输出长度、请求复杂度和 SLA 时延要求都在持续变化，边缘节点又常常缺乏云侧大规模请求池，静态 batching、固定 batch size 和固定服务路径难以稳定满足业务目标。对于移动用户，接入点和最优服务实例还会随位置变化而变化，最近节点不一定是负载最合适或端到端时延最低的节点；若用户会话中的 KV 或上下文状态仍保留在旧边缘节点，服务切换还可能引发状态迁移、远程访问或重新 prefill。

因此，CEC LLM inference 的挑战并不是单一技术点，而是模型部署、batching 调度、task offloading、状态管理和服务路由之间的耦合。模型切分、prefill/decode 分离、KV 迁移、跨节点执行和多入口路由都可能带来新的 activation、hidden state、KV Cache 或 session state 传输成本。系统需要在计算收益和通信代价之间持续权衡，并在 QoS、隐私和服务连续性之间给出可解释的在线决策。

\subsection*{1.3 国内外研究现状概述}
\addcontentsline{toc}{subsection}{1.3 国内外研究现状概述}

现有研究为 CEC LLM inference 提供了多方面基础。一方面，on-device inference 关注模型压缩、低比特执行、小模型设计和端侧硬件加速，使部分请求能够在用户设备本地完成；FlexGen、Petals、HexGen 等工作则进一步展示了资源受限或异构环境下的大模型推理可以通过分层存储、协作式 block 放置和异构并行来突破单节点容量限制 \refcite{1}, \refcite{2}, \refcite{3}。另一方面，cloud/datacenter LLM serving 已经围绕 continuous batching、PagedAttention、chunked prefill、prefill/decode disaggregation、KV cache management 和多实例调度形成了较成熟的系统路线，Orca、vLLM、Sarathi-Serve、DistServe、SGLang 和 Mooncake 等系统分别代表了不同层面的关键机制 \refciterange{4}{8}, \refcite{21}。

边缘计算领域长期研究 task offloading、资源分配、移动性管理和网络感知调度，但传统 MEC/CEC 任务通常被抽象为给定输入大小、计算量、deadline 和能耗约束的 job。LLM inference 改变了这一抽象：推理时间取决于 prompt、输出长度、decode step、batch 状态和 KV 状态，复杂请求还可能涉及多模型、多 agent 或多阶段执行。因此，CEC LLM inference 位于 LLM serving systems 与 CEC task offloading 的交叉处，既要继承 LLM serving 中的 prefill/decode、KV Cache、batching、speculative decoding 和分布式执行问题，又要处理端-边-云资源异构、动态网络、用户移动、隐私约束和服务连续性。

基于这一交叉特征，本文将相关工作组织为三条主线。第一条是 \textbf{CEC 下的资源自适应模型部署方法}，关注模型结构、模型组件、执行阶段和生成职责如何在端、边、云之间切分、放置和重配置；这一方向直接对应后文的 layer-wise / tensor-level / expert-level / phase-level / semantic-token-level 等切分范式。第二条是 \textbf{CEC 中的业务感知 batching 参数优化算法}，关注请求进入推理实例后如何根据 SLO、队列状态、KV 占用和 prefill/decode 干扰选择 batch 形成与更新策略。第三条是 \textbf{CEC 中的 LLM inference task offloading}，关注完整请求、会话、multi-agent workflow 以及 context/KV/runtime state 如何在端、边、云之间路由、卸载、迁移和复用。这三条主线分别从空间部署、时间调度和服务路径三个角度刻画 CEC LLM inference 的核心优化问题。

\subsection*{1.4 本文研究范围与主要贡献}
\addcontentsline{toc}{subsection}{1.4 本文研究范围与主要贡献}

本文聚焦 \textbf{CEC 场景下的大语言模型推理优化问题}。研究范围限定在推理阶段，不讨论大模型训练、预训练或微调过程；重点关注端侧、接入边缘、区域边缘和中心云之间的大模型推理协同，包括模型动态切分、自适应 batching、prefill/decode 调度、分层 KV/状态管理、任务 offloading、策略路由、移动性感知请求调度和服务连续性保障。


对于底层 kernel 优化、硬件实现和模型结构设计，本文仅在其影响 CEC 推理系统设计时进行讨论。例如，低比特 kernel 会影响模型能否部署在端侧或边缘设备上；KV Cache layout 会影响状态迁移和缓存复用；partition-friendly model design 会影响模型是否适合跨端-边-云切分执行。本文的重点不是单独评估某个 kernel 或模型结构，而是从 CEC 系统角度分析这些技术如何影响大模型推理的部署、调度、状态管理和服务策略。

本文首先从 prefill/decode、KV Cache、batching/scheduling、decoding runtime 和主要服务指标出发，梳理 LLM inference 的基本执行机制，并进一步解释模型规模、KV 状态、请求动态性、decode 串行性和多实例协同为何会成为推理系统的核心瓶颈。在此基础上，本文构建面向优化对象的 Layer–Scope taxonomy，将 LLM inference optimization 划分为 Model-Level、Kernel-Level、Runtime-Level 和 Serving Policy \& Routing-Level，并以 Local / Distributed 作为执行范围副轴，用于分析不同方法的 primary optimization object、execution scope 和 secondary effects。

随后，本文分析 CEC 环境如何改变 LLM inference 的优化重心。相较云侧集中式 serving，CEC 场景需要额外处理端-边-云资源异构、边缘低并发、动态网络、用户移动、状态迁移、隐私边界和 QoS 约束，因此各优化层都需要从单点效率转向跨节点、跨状态、跨服务路径的协同决策。围绕这一变化，本文提出三类面向 CEC 的综述主线：资源自适应模型部署方法、业务感知 batching 参数优化算法和 LLM inference task offloading，并分别总结其核心问题、代表工作、决策变量、优化目标和面向验证环境的算法建议。最后，本文从时延、吞吐、QoS、资源利用率、通信开销、状态迁移成本、隐私约束、服务连续性和系统可部署性等角度归纳评估维度，并讨论跨层协同优化、状态感知 routing、移动性预测、异构 runtime、分层模型服务和真实系统落地等未来挑战。
